# Phase 0 implementation — corrections caught while building :

- Wrote `src/middleware.ts` following the long-standing Next.js convention (from training data). The dev server itself flagged it as deprecated: Next.js 16 renamed the convention to `src/proxy.ts`, exporting a function named `proxy` instead of `middleware` (the `config`/`matcher` export is unchanged). Caught by actually running the dev server and reading its warning, not by memory — consistent with CLAUDE.md's "always fetch the latest documentation" rule. Fixed by renaming the file and the exported function.
- Split `src/lib/auth.ts` into an Edge-safe `src/lib/session.ts` (jose + cookies only) and `src/lib/auth.ts` (adds the Mongoose-dependent `getCurrentManager()`). The proxy/middleware file runs on the Edge runtime, which can't load Mongoose — importing the combined file from `proxy.ts` would have broken at runtime even though it type-checked fine.
- `@getbrevo/brevo` v6 is a fully rewritten SDK (different from the older `SendinBlue`/v2-style `TransactionalEmailsApi` API remembered from training) — read the package's own `.d.ts` files under `node_modules` to get the real `BrevoClient`/`sendTransacEmail` shape instead of guessing.
- Mongoose model typing: `models.Manager ?? model("Manager", schema)` combined with `InferSchemaType` produced a `never`/malformed union type on reads (e.g. `Manager.findById().select()` typed as possibly an array). Fixed by declaring an explicit `IManager` interface and casting the `models.X` branch to `Model<IManager>`, which is the standard pattern for Mongoose + Next.js + TypeScript.
- No local MongoDB/Redis/Docker/chromium-cli/Playwright available in this sandbox, so Phase 0's DB-backed flows (actual register/login/OTP) could not be exercised end-to-end at first — only static rendering, the `/dashboard` redirect, lint, and typecheck were verified. Flagged explicitly in PROGRESS.md rather than claiming full verification.

# Phase 3 — Page CRUD & Blocknote content :

- Before writing any code, checked Blocknote's actual current docs (v0.51.4) instead of assuming from training data — good thing I did, because two things already written into `docs/PRE_BUILD_PLAN.md`'s schema (from the very first pre-build review, before I knew how Blocknote actually works) turned out to be wrong:
  - `Content` was speced as one document *per block*, chained via `prevContent`/`nextContent`. Blocknote's real document shape is a single JSON array (`editor.document`), set/read as one unit via `initialContent`/`onChange` — ordering and nesting already live in the array + each block's own `children`. The linked-list design would have meant constantly translating between that shape and Blocknote's own on every keystroke, for a "block-independence" property Blocknote already gives you natively. Replaced with one `Content` document per `Page` holding the whole array.
  - `Page.nextPage` was a singly-linked chain (one child per page). The spec actually wants a general tree — "nested pages **in** a page" (plural) and the dashboard needs a page tree, not a chain. Replaced with `parentPageId`.
  - Both caught and fixed *before* implementation, not after — the whole reason for checking the library's docs first instead of building against the original (wrong) schema and discovering the mismatch mid-Phase-3.
- This also meant correcting the Tech Stack line about real-time collaboration: it said "WebSockets via a dedicated Node/Socket.io server," but Blocknote's collaboration requires a **Yjs document** + a Yjs-compatible provider (`y-websocket`/Hocuspocus) — Socket.io doesn't speak Yjs's binary sync protocol, you'd have to hand-roll that on top of it. Corrected in PRE_BUILD_PLAN.md now, ahead of actually planning Phase 4, so Phase 4 doesn't start from a wrong premise either.
- Chose `@blocknote/shadcn` over `@blocknote/mantine` for the editor theme — checked both packages' peer dependencies on npm before deciding. Mantine is a full separate component/CSS framework; shadcn peer-depends on `tailwindcss@^4` directly and bundles Radix + `lucide-react` (already a dependency), so it doesn't fight the CLAUDE.md "Tailwind CSS" UI rule the way Mantine would have.
- **Real bug, not just a code-review catch**: `useCreateBlockNote` throws `ReferenceError: window is not defined` when server-rendered — lint and `tsc --noEmit` were both clean (this isn't a type error, it's a runtime environment mismatch), and it only surfaced when I actually navigated to the page editor route during live testing and read the 500 in the server log. Fixed with a `next/dynamic(..., { ssr: false })` client-only wrapper (`src/components/PageEditorClient.tsx`) — `ssr: false` can't be used directly inside a Server Component (Next.js App Router restriction), hence the separate wrapper file rather than inlining the option where the component is used. This is exactly the kind of thing "verify by running the app, not just reading the code" is for.
- Extended `getOwnedWorkspace` (Phase 1) into `getAccessibleWorkspace` (manager-or-employee) rather than writing a parallel check — Page routes reuse it via `src/lib/pages.ts`, keeping one place that defines "who can access this workspace."
- Verified employee-side access control without real OAuth credentials by seeding an `Employee` + `WorkspaceMember` directly and hand-signing a session JWT with the same `jose` secret the app already uses — legitimate way to test the authorization logic (`getAccessibleWorkspace`'s employee branch) in isolation from the OAuth handshake itself, which is what's actually blocked on missing credentials.

# Phase 4 — Real-time collaboration (Yjs + Hocuspocus) :

- Checked Hocuspocus's actual current docs (v4.4.0) before designing, same as Blocknote/Brevo/Next.js 16 earlier — its `onAuthenticate`/`onLoadDocument`/`onStoreDocument` hooks map directly onto what this app needs, which is why it was chosen over hand-rolling auth/persistence on top of raw `y-websocket`.
- Asked the user two explicit questions before writing the plan, since both are real product/ops tradeoffs I shouldn't decide unilaterally: (1) should Yjs collaboration replace the Phase 3 autosave entirely, or stay as an optional layer with a fallback, and (2) is running a second long-running process (locally and in production) acceptable. Both confirmed "yes, the simpler option" — recorded here since it explains why `PUT /api/pages/[id]/content` was deleted outright rather than kept as a fallback.
- **Real bug, caught only by actually running `npm run dev:collab`**, not by lint or `tsc --noEmit`: importing `@blocknote/server-util` from `server/collaboration.ts` crashed at startup (`TypeError: h.default.extend is not a function`), because the project has no `"type": "module"` in `package.json`, so `tsx` resolved the package's CJS build, which has a Tiptap ESM/CJS interop bug. Fixed by renaming the file to `server/collaboration.mts` — `.mts` forces ESM resolution for that one file regardless of the rest of the project's module type, which is a more targeted fix than flipping the whole project to `"type": "module"` (a much bigger blast radius for a problem confined to one standalone script).
- **Testing without a browser, done rigorously rather than skipped**: `HocuspocusProvider` and `Y.Doc` are plain JS/Node-compatible classes, not React-specific, so I could write real Node scripts that connect to the running collaboration server exactly like a browser client would — this is a legitimate test, not a mock. Verified: collab-token auth (401/404/200), the bootstrap-from-legacy-JSON migration, and genuine multi-client real-time sync (a `Y.Map` change from one connection observed by a second connection within ~100ms), plus that persistence actually wrote `yjsState`/`blocks` to Mongo after disconnecting.
- One thing this approach could *not* verify: I hand-crafted a `Y.XmlElement`/`Y.XmlText` to simulate a "real" Blocknote content edit, and it didn't round-trip through `yDocToBlocks` correctly. Rather than assume this meant the sync was broken, I isolated the variable — a separate raw `Y.Map` test confirmed the relay itself works perfectly, so the XML edit's failure was almost certainly because I don't know Blocknote's exact internal ProseMirror/y-prosemirror node shape well enough to construct one by hand (which only a real editor instance produces correctly). Recorded as a known, honest gap rather than papered over: real "type in one tab, see it in another" behavior still needs a human with a real browser.
